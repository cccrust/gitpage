# v1.3 — SSH Info API, Container Resource Limits, Health Check Retry

## Changes

### Backend

- **Removed unused `staging_root`** field from `DockerManager`
- **Resource limits**: `memory_limit` (parse "1g"/"512m" via `parse_memory_limit()`) and `cpu_shares` from `[docker]` config applied to container `HostConfig`
- **Container sshd**: per-user container CMD creates user, sets random 12-char password via `chpasswd`, starts `/usr/sbin/sshd -D` alongside `sleep infinity`
- **Password generation**: `generate_password(12)` → random alphanumeric; stored in `password_allocations: Arc<Mutex<HashMap<String, String>>>`; retrieved via `get_user_ssh_password()`
- **Health check retry**: `exec_check_status` retries 10×500ms before reporting app not running
- **SSH info API** (`src/docker.rs`): `get_user_ssh_port(username)` → `Option<u16>`, `get_user_ssh_password(username)` → `Option<String>`
- **New endpoint**: `GET /api/user/ssh-info` → `{ mode, ssh_port?, ssh_password?, container? }` in `handlers/auth.rs:ssh_info`
- **Background restore**: `restore_apps_on_startup` runs via `tokio::spawn` so server starts immediately without waiting for container health checks

### Frontend

- **New page**: `/docker-status` (`DockerStatusPage.tsx`) shows mode, container name, SSH port, SSH password, connection command with `ssh -p <port> root@localhost`
- **Nav link**: "Docker" added to topnav (logged-in users)
- **API function**: `getSshInfo()` returns `SshInfo` interface

### Test

- **`test_docker_mode.sh`**: step 11a now retries up to 5×1s (instead of single call) to wait for background restore to complete
