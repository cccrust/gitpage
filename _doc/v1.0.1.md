# v1.0.1 — Organization 組織功能

讓使用者可以建立類似 GitHub Organization 的組織，組織擁有自己的 container、repo、成員管理。

## 動機

- 多使用者協作同一專案時，需要共享的命名空間
- 組織未來可以被視為一個「個體」——擁有自己的 Docker container、SSH 存取、部署環境
- 為 v1.1 容器化做準備：每個組織（如同每個使用者）會有一台專屬開發容器

---

## 資料模型

### `organizations` 表

```sql
CREATE TABLE organizations (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL UNIQUE,          -- 唯一識別名 (orgname)
    display_name TEXT DEFAULT '',
    description TEXT DEFAULT '',
    owner_id    INTEGER NOT NULL REFERENCES users(id),  -- 建立者
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### `organization_members` 表

```sql
CREATE TABLE organization_members (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    org_id      INTEGER NOT NULL REFERENCES organizations(id),
    user_id     INTEGER NOT NULL REFERENCES users(id),
    role        TEXT NOT NULL DEFAULT 'member',  -- 'admin' | 'member'
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(org_id, user_id)
);
```

### `repositories` 表修改

新增 `owner_type` 與 `org_id` 欄位，支援雙重擁有者：

```sql
ALTER TABLE repositories ADD COLUMN owner_type TEXT NOT NULL DEFAULT 'user';
ALTER TABLE repositories ADD COLUMN org_id INTEGER REFERENCES organizations(id);
```

- `owner_type = 'user'` → 沿用現有 `user_id`
- `owner_type = 'org'` → `user_id` 為建立者，`org_id` 指向組織
- 遷移腳本：所有現有 repo 設 `owner_type = 'user'`

### `pages_config` / `apps_config` / `ssh_keys` / `deploy_logs`

這些表透過 `repo_id` 關聯到 repositories，而 repositories 已知道 owner，故不需改 schema。查詢時透過 JOIN 即可區分組織/使用者倉庫。

## API

### 組織 CRUD

| 方法 | 路徑 | 說明 |
|------|------|------|
| POST | /api/orgs | 建立組織 |
| GET | /api/orgs | 列出我的組織 |
| GET | /api/orgs/:name | 組織資訊 |
| PUT | /api/orgs/:name | 更新組織（admin only） |
| DELETE | /api/orgs/:name | 刪除組織（owner only） |

### 成員管理

| 方法 | 路徑 | 說明 |
|------|------|------|
| GET | /api/orgs/:name/members | 成員列表 |
| POST | /api/orgs/:name/members | 新增成員（admin） |
| DELETE | /api/orgs/:name/members/:user_id | 移除成員（admin） |
| PUT | /api/orgs/:name/members/:user_id | 修改角色（admin） |

### 組織 Repo

| 方法 | 路徑 | 說明 |
|------|------|------|
| GET | /api/orgs/:name/repos | 列出組織的 repos |
| POST | /api/repos?org=:name | 在組織下建立 repo |

### 現有 API 相容

`GET /api/users/:username/repos` → 僅回傳使用者個人 repos
`GET /api/orgs/:name/repos` → 回傳組織 repos

## 權限模型

| 角色 | 權限 |
|------|------|
| **Owner** | 刪除組織、管理所有成員、所有 repo 完整權限 |
| **Admin** | 更新組織設定、管理成員、組織 repo 管理權限 |
| **Member** | 組織 repo 的 push/pull、建立/編輯 repo、deploy |

目前 `ssh_keys` 已綁定 `(user_id, repo_id)`，不需修改模型：
- 使用者在組織 repo 加入自己的 SSH key → 可 SSH 進自己帳號的容器操作該 repo
- v1.1 容器化後：組織也會有自己的容器，admin/member 可 SSH 進組織容器

## 前端頁面

| 路由 | 頁面 |
|------|------|
| /orgs | 組織列表（我的組織） |
| /orgs/new | 建立組織 |
| /org/:name | 組織首頁（成員 + repos） |
| /org/:name/settings | 組織設定 |
| /org/:name/members | 成員管理 |
| /org/:name/repos/:repo_id/... | 組織下的 repo 頁面（與個人 repo 共用元件） |

組織 repo 的路由維持 `/repo/:id`，但顯示時加上組織名稱標示。

## 實作順序

```
1. 資料庫遷移 — organizations + members 表，repositories 加欄位
2. 後端 API — org CRUD + members CRUD
3. 後端修改 — repo 建立/查詢支援 org 參數
4. 遷移腳本 — 現有資料無痛升級（owner_type = 'user'）
5. 前端頁面 — 組織列表、建立、設定、成員管理
6. 前端修改 — repo 頁面顯示組織資訊
7. SSH key — 組織 repo 的 SSH key 管理
```

## 與 v1.1 Docker 的關係

```
v1.0.1                    v1.1
使用者 Alice              容器 gitpage-alice
  ├─ 個人 repos            /home/alice/projects/
  └─ 組織 MyTeam ──────── 容器 gitpage-org-myteam
      └─ 組織 repos          /home/myteam/projects/
```

- 每個組織在 v1.1 會獲得一台專屬容器，類似使用者的容器
- 組織容器的 SSH 可由 admin/member 存取
- 組織的 repo 在容器內位於 `/home/{orgname}/projects/` 下
- 組織容器也共用同一個 `gitpage/dev` 基底映像檔
