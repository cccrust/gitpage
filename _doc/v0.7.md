# v0.7 — Dropbox-like File Manager（免 git 操作）

讓使用者可以不需學習 git，直接在瀏覽器中創建專案、拖放上傳檔案、編輯與刪除。所有變更先暫存在 working tree，使用者準備好後按下「儲存版本」才 commit。

---

## 架構概覽

```
使用者操作 (Drag & Drop / Create / Edit / Delete)
       ↓
  修改 working tree（不 commit）
       ↓
  ┌─────────────────────────────┐
  │  使用者按下「儲存版本」       │
  │  填寫 commit message        │
  └─────────────────────────────┘
       ↓
  git add + git commit（一次批次）
       ↓
  auto_deploy_pages / auto_deploy_app
```

核心概念：working tree 相當於 Dropbox 的本地資料夾，隨時可改；「儲存版本」相當於按一下同步 + 建立版本紀錄。

---

## 1. Working Tree 管理

### 目錄結構

```
data/repos/{username}/{repo}.git/   ← bare repo (git 物件)
data/repos/{username}/{repo}/       ← working tree (可編輯的檔案)
```

現有 Pages 與 Apps deploy 已經使用 `git --work-tree` checkout 到 `source/` 目錄，但那是唯獨的 deploy 用副本。v0.7 新增一組**可直接寫入的 working tree**，路徑沿用既有 `data/repos/{username}/{repo}/`（與 bare repo 同層）。

### 初始化

建立 repo 時（`created_via=web`）：
1. 建立 bare repo
2. checkout 出 working tree（含初始 commit，一個空 commit 或 README.md）

### 狀態追蹤

前端需要知道哪些檔案相對於上次 commit 有變更：

```
git status --porcelain
 M src/index.html     → 已修改
?? new-file.txt       → 新增
 D old-file.txt       → 刪除（已從 working tree 移除）
```

後端提供 API 查詢變更狀態，讓前端顯示「未儲存變更」提示。

## 3. API

### Repo CRUD（擴充）

| 方法 | 路徑 | 說明 |
|------|------|------|
| POST | /api/repos | 新增 `created_via` 參數，可選 `template` |

建立時若 `created_via=web`，自動：
1. 建立 bare repo
2. checkout 出空的 working tree（含初始 commit）
3. 若指定 template，複製範本檔案

### File API（操作 working tree，不 commit）

| 方法 | 路徑 | 說明 |
|------|------|------|
| GET | /api/repos/:id/tree?path=/ | 列出目錄內容 |
| GET | /api/repos/:id/raw?path=/README.md | 取得檔案內容 |
| PUT | /api/repos/:id/files?path=/index.html | 建立/覆寫檔案（body 為檔案內容） |
| DELETE | /api/repos/:id/files?path=/old.txt | 刪除檔案 |
| POST | /api/repos/:id/upload | 多檔案上傳（multipart/form-data）|
| POST | /api/repos/:id/mkdir?path=/newdir | 建立目錄 |
| POST | /api/repos/:id/move?from=&to= | 移動/重新命名 |
| GET | /api/repos/:id/status | 列出未 commit 的變更 |

### Commit API（批次儲存版本）

| 方法 | 路徑 | 說明 |
|------|------|------|
| POST | /api/repos/:id/commit | 提交所有未儲存變更 |

### POST /api/repos/:id/commit request

```json
{
  "message": "Add landing page and upload assets"
}
```

後端執行：
1. `git add -A`（所有變更）
2. `git commit -m "{message}"`
3. 觸發自動部署（若 pages / apps 有啟用）

### GET /api/repos/:id/status response

```json
{
  "pending": true,
  "changes": [
    { "path": "src/index.html", "type": "modified" },
    { "path": "new-file.txt",   "type": "added" },
    { "path": "old-file.txt",   "type": "deleted" }
  ],
  "last_commit": {
    "message": "Initial commit",
    "author": "gitpage",
    "date": "2026-06-16T12:00:00Z"
  }
}
```

### 檔案操作（不 commit）

```rust
pub fn write_file(repo_path: &str, file_path: &str, content: &[u8]) -> Result<()> {
    let full_path = format!("{}/{}", repo_path, file_path);
    std::fs::create_dir_all(Path::new(&full_path).parent().unwrap())?;
    std::fs::write(&full_path, content)?;
    Ok(())
}
```

### Commit 操作（批次）

```rust
pub fn commit_all(repo_path: &str, message: &str, username: &str) -> Result<()> {
    let repo = git2::Repository::open(repo_path)?;
    let mut index = repo.index()?;
    index.add_all(["*"], git2::IndexAddOption::DEFAULT, None)?;
    index.write()?;
    let tree_id = index.write_tree()?;
    let tree = repo.find_tree(tree_id)?;
    let parent = repo.head()?.peel_to_commit()?;
    let sig = git2::Signature::now(username, "gitpage@localhost")?;
    repo.commit(Some("HEAD"), &sig, &sig, message, &tree, &[&parent])?;
    Ok(())
}
```

## 6. 部署整合

### 觸發時機

只有當使用者按下「儲存版本」（commit）時，才觸發自動部署：

```
POST /api/repos/:id/commit
  → git add -A
  → git commit -m "..."
  → auto_deploy_pages()   (若 pages enabled)
  → auto_deploy_app()     (若 apps enabled)
```

### 與 git push 共存

既有的 git push 流程完全不受影響：
- git push → bare repo 更新 → auto_deploy（既有）
- Web UI 操作 → working tree 更新 → commit → auto_deploy（新增）

兩條路徑最終都透過 bare repo 的 commit 來觸發部署。

---

## 7. 安全性

- 所有 file API 需驗證：僅 repo owner 可寫入
- 路徑 traversal 防護：`path` 不得包含 `..`
- 檔案大小限制：單檔 max 10MB（configurable）
- commit message 不可為空

## 9. 注意事項

- working tree 路徑與 git push 共用同一目錄，確保兩者操作不互相干擾
- 大量小檔案操作後 commit 可能會慢（`git add -A` 掃整個目錄）
- 前端定時（或手動）檢查 `GET /api/repos/:id/status` 更新變更狀態
- 切換分支前需提醒使用者先儲存版本
