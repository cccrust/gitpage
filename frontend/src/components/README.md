# Components

## Overview

The `components/` directory contains four shared UI components used across multiple pages. Each component is a single-purpose, stateless (or minimally stateful) React component that encapsulates a specific rendering pattern. There is no component library, no design system, and no atomic design methodology — each component is written ad-hoc for its specific use case.

---

## Layout (`Layout.tsx`)

### Purpose

`Layout` is the navigation shell that wraps every page in the application. It renders a top navigation bar, a main content area (via `children`), and a bottom navigation bar visible on mobile.

### Conditional Rendering Based on Login State

The component calls `isLoggedIn()` on every render to decide which navigation links to show:

- **Anonymous state** — shows "Login" and "Register" links in both top and bottom nav bars.
- **Authenticated state** — shows "Repos", "Orgs", "Docker", "Settings" links in the top nav, plus "Logout" (which calls `clearToken()` and navigates to `/login`).

This is a synchronous, imperative check — there is no auth context subscription. If the token is removed from localStorage while the user is on a page, the nav will not update until the next render cycle (navigation or state change).

### Architecture

- Uses `useLocation()` to read the current path for bottom nav active-link highlighting.
- Uses `useNavigate()` to perform programmatic navigation after logout.
- Renders two separate `<nav>` elements: one with class `topnav` and one with `bottom-nav`. The bottom nav is styled to appear fixed at the bottom of the viewport on small screens.
- The logo is a `<Link to="/">gitpage</Link>` that always points home.

## Spinner (`Spinner.tsx`)

### Purpose

A minimal loading indicator component. It renders a `<div className="loading">` with configurable text.

### Usage

```tsx
<Spinner />                    // renders "Loading..."
<Spinner text="Searching..." /> // renders "Searching..."
```

### Design

- Accepts a single optional prop: `text` with a default value of `"Loading..."`.
- Does not use any CSS animation, spinner graphic, or progress bar — just plain text in a styled div.
- Returns `null`-equivalent content: a simple `<div>` with text. The parent component controls visibility by conditionally rendering `<Spinner />` based on a `loading` state variable.
- There is no timeout, no error boundary, and no retry mechanism. The spinner is purely a visual indicator that data is being fetched.

This simplicity is deliberate — the project avoids external dependencies for loaders. The CSS class `loading` can be styled globally in `index.css` to add animations (e.g., a pulsing text or a CSS spinner) without changing the component code.

## MarkdownView (`MarkdownView.tsx`)

### Purpose

Renders pre-rendered HTML (generated by the backend from Markdown) into the DOM, then enhances it with client-side KaTeX math rendering and Mermaid diagram rendering.

See the separate `MarkdownView.md` for a detailed architectural analysis.

### Key Props

- `html: string` — the HTML string returned by the backend's `getReadme` or `getBlob` endpoints. This is already rendered Markdown, not raw Markdown text.

## Pagination (`Pagination.tsx`)

### Purpose

A simple offset-based pagination control with Previous and Next buttons and a "current / total" label.

### Props

| Prop | Type | Description |
|------|------|-------------|
| `page` | `number` | Current page number (1-indexed) |
| `totalPages` | `number` | Total number of pages |
| `onPageChange` | `(page: number) => void` | Callback fired when the user clicks Prev or Next |

### Behavior

- **Hides itself** when `totalPages <= 1` — there is nothing to paginate.
- **Disables Prev** when `page <= 1`.
- **Disables Next** when `page >= totalPages`.
- Renders a flex container with three children: a Prev button, a `"page / totalPages"` span, and a Next button. The styling is applied inline (`style={{ display: 'flex', gap: 8, justifyContent: 'center', marginTop: 16 }}`) rather than through a CSS class.

### Usage in the Application

`Pagination` is imported by the `Dashboard` page's search results section. The `Dashboard` manages its own search page state (`searchPage`, `searchTotalPages`) and passes them to `Pagination` alongside a `doSearch()` callback that performs the actual API fetch. This follows the **controlled component** pattern — the parent owns the state, and `Pagination` is a pure presentation component.

The component does not:
- Support page number input (no direct page jump)
- Show ellipsis or page number buttons (no page range display)
- Handle keyboard navigation
- Manage its own state

This simplicity is appropriate for the project's pagination needs — search results and repo lists rarely exceed a few pages, and users navigate sequentially through results.
